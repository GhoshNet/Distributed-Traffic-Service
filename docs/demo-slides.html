Problem Statement

What are we building?

The Constraint

No driver may start a journey without a confirmed booking. The system must enforce this even under concurrent load and partial failures.

The Scale

1M active drivers nationally. 30% book in a 30-minute morning peak = ~167 bookings/s. Our two-node demo targets 50/s per node.

Why database-per-service?

Independent schema evolution, no cross-service joins, bounded failure blast radius. Each service owns its data contract. Trade-off: joins become API calls.

Why every laptop runs the full stack?

Losing one laptop doesn't take the system down. No shared database tier to fail. Active-active means any node can serve any request. Trade-off: more memory per machine.

Transactional Outbox

Journey row and outbox event written in one DB transaction. Background drainer publishes to RabbitMQ every 2s. Survives broker restarts — no confirmed booking is ever lost.

Idempotency Keys

Client sends Idempotency-Key header. Duplicate requests return the cached result without re-running the saga. Safe to retry on network failure.

Conflict Check

Three checks in one SERIALIZABLE transaction: driver overlap, vehicle overlap, road-segment capacity. SELECT FOR UPDATE prevents two concurrent bookings both passing.

Conflict Detection

Grid-cell road capacity model

How the grid works

Each road is broken into ~1km grid cells (0.01° resolution). Each cell has a capacity counter per 30-minute time slot. A booking walks the actual waypoints of its route and increments only the cells it physically crosses.


Two routes that are geographically close but on different roads never contend — the lock is scoped to the exact cells traversed.

Why SERIALIZABLE + SELECT FOR UPDATE?

Application-level locking (Redis SETNX, optimistic version columns) has a race window between read and write. Two concurrent bookings for the same cell and slot will both read "capacity available" before either writes.


SERIALIZABLE + SELECT FOR UPDATE means the second transaction deadlocks and retries, or sees the first commit and rejects. No race window at all.

Multi-Node Design

How two laptops talk to each other

Peer Discovery

Static env vars (PEER_CONFLICT_URLS) for known nodes at startup. Dynamic gossip via POST /admin/peers/register at runtime — propagates to conflict, user, and journey services automatically.

Journey Replication

After every confirmed booking, a goroutine pushes to all peers via POST /internal/journeys/replicate. On boot: full catch-up pull from each peer. Safety net: 5-minute periodic re-sync.

Slot Replication (Conflict Service)

After each successful conflict check, slots are pushed to peers via /internal/slots/replicate. Catch-up sync on boot. Result: a booking on Node A blocks the same slot on Node B (after replication propagates).

Health Model: ALIVE → SUSPECT → DEAD

Each node probes peers every 10s. 3 misses → SUSPECT. 6 misses → DEAD. <50% alive → LOCAL_ONLY mode. Visible at /health/nodes and in the frontend dashboard.

Key insight: JWTs are signed with the same shared secret on all nodes. A session started on Laptop A is valid on Laptop B — no re-login after failover.

Design Decisions & Tradeoffs

Why we made the choices we did

Decision Why we chose it The tradeoff we accepted
Active-active (every laptop runs full stack) Losing one node doesn't require manual failover. Any node can serve any request. Two bookings submitted simultaneously to two nodes can both pass within the ~200ms replication window. Documented, accepted for demo scale.
Transactional outbox over dual-write Eliminates "wrote to DB, crashed before publishing to RabbitMQ" — the most common event-loss bug. Extra table to poll, 2-second delivery lag to RabbitMQ. Worth it for the durability guarantee.
SERIALIZABLE in conflict-service only Only the conflict check needs serializable — ordinary CRUD (user login, journey list) doesn't. Avoids locking overhead on hot paths. More complex reasoning about which operations are safe at READ COMMITTED.
Saga pattern (not distributed 2PC everywhere) Saga is non-blocking. 2PC blocks both participants until the coordinator responds — bad under network partitions. Requires explicit compensating transactions. 2PC is offered as an opt-in demo path (?mode=2pc).
Redis-first enforcement Roadside checks are read-heavy and latency-critical (<200ms hard limit). Redis gives <20ms p95 on cache hit. Cache is cold on boot. First check after restart always pays the fallback REST cost.
Shared JWT secret across nodes Enables transparent session migration — the user doesn't re-login after failover to a peer node. No per-node token revocation. Compromised secret affects all nodes.
Honest Assessment

Known limitations — not hidden

Cross-node double-booking window
Two bookings to two different nodes within ~200ms can both be CONFIRMED for the same slot. Fixing this requires a consensus protocol (Paxos/Raft) across nodes — out of scope for this exercise. Documented trade-off of active-active eventual consistency.
No saga compensation retry
If the conflict-service is permanently unreachable, the booking fails fast with REJECTED. There is no automatic retry loop or dead-letter compensation.
Enforcement cache cold on boot
First enforcement check after a restart always misses Redis and pays the fallback REST cost. No cache warming on startup was implemented.
WebSocket registry in memory only
The notification service holds active WebSocket connections in a process-local map. A restart drops all live connections; clients must reconnect. Notification history is safe in Redis — no data is lost, only the live push.
No token blacklisting
Revoked JWTs remain valid until natural expiry (24h). No deny-list on logout. Acceptable for a demo system; production would need Redis-backed revocation.
RabbitMQ cluster unreliable on single host
Three RabbitMQ nodes on one Docker host compete for the Erlang distribution port. All messaging semantics are demonstrated; cluster HA is better shown across physical machines.
