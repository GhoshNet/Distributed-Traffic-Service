Journey Booking System
How It Works

A distributed, fault-tolerant system where drivers pre-book road journeys before setting off. This document explains every part of the system — what it does, why it was designed that way, and what tradeoffs were made.

Microservices Saga Pattern Transactional Outbox Circuit Breaker SERIALIZABLE Isolation Redis Sentinel Active-Active RabbitMQ WAL Replication

1Overview — What is this system?

Imagine a country where no driver can get in their car without having a pre-approved booking. The booking tells the system where you're going, when you're leaving, what vehicle you're driving, and what roads you'll use. An enforcement officer can check your plate at the roadside and instantly know if you're supposed to be there.

This system handles that. Its core challenge is: how do you check and reserve a road slot atomically, across multiple servers, without creating double bookings — even when parts of the system are down?

What drivers can do

Register an account, add their vehicle, book a journey (specifying origin, destination, departure time, route), receive a real-time confirmation or rejection, cancel a booking, check their points balance.

What enforcement agents can do

Look up any vehicle plate or driving licence number and get a sub-second response: is this vehicle supposed to be on the road right now? The system returns the journey details if valid, or "no valid booking" if not.

Scale the system is designed for

1 million active drivers nationally. Assuming 30% book during a 30-minute morning peak, that's roughly 167 bookings per second. Our two-laptop demo targets 50 bookings/second per node. Enforcement checks are high-frequency reads and must return in under 200ms — ideally under 20ms from cache.

2The 6 Services

The system is split into six independent services. Each one owns its own database and can fail, restart, or scale without directly affecting the others. They communicate via REST APIs for real-time operations and via a message broker (RabbitMQ) for background work.

👤 User Service
Python (FastAPI) · Port 8001
DB: users_db

Handles registration, login, and vehicle management. Passwords are hashed with bcrypt. On login, it issues a JWT token signed with a shared secret — the same secret is used on every node, so a token issued on Laptop A is valid on Laptop B with no re-login.

Read queries go to a Postgres replica; writes go to the primary. Before inserting a new user, it acquires a distributed lock across all peer nodes to prevent two nodes simultaneously accepting the same email address.

🗺 Journey Service
Python (FastAPI) · Port 8002
DB: journeys_db

The most complex service — it orchestrates the booking saga. When a driver books, this service coordinates the entire flow: checking for duplicates, calling the Conflict service to reserve a road slot, saving the confirmed booking, and writing an event for downstream services. It also runs background tasks: a lifecycle scheduler that transitions journeys from CONFIRMED → IN_PROGRESS → COMPLETED, and an outbox drainer that publishes events to RabbitMQ.

⚡ Conflict Service
Go · Port 8003
DB: conflicts_db

The gatekeeper. When asked "can this booking proceed?", it runs three checks atomically inside a single SERIALIZABLE database transaction with SELECT FOR UPDATE: (1) is the driver already booked at this time? (2) is the vehicle already booked? (3) is the road at capacity? All three must pass for the booking to be approved.

Road capacity is modelled as a grid of ~1km cells, each with a counter per 30-minute time slot. The lock only covers the cells the route actually passes through, so two bookings on different roads never contend.

🔔 Notification Service
Go · Port 8004
Store: Redis

Listens for booking events from RabbitMQ and pushes them to the driver's browser in real-time via WebSocket. The connection map is held in memory (one entry per logged-in user). Notification history is stored in Redis as a capped list (50 entries, 7-day TTL) so drivers can see missed notifications after a reconnect.

It deduplicates messages using Redis SETNX on the message ID, so redelivered events are silently ignored. Unprocessable messages go to a dead-letter queue (DLQ) with a 24-hour TTL for inspection.

🚔 Enforcement Service
Python (FastAPI) · Port 8005
Store: Redis + Journey API

Built for speed. When an enforcement agent queries a vehicle plate, it checks Redis first (sub-20ms). On a cache miss, it falls back to calling the Journey Service REST API (under 100ms). The cache entry is populated when a booking is confirmed (via RabbitMQ event) and invalidated when a journey is cancelled or completed.

Licence-to-user-ID mappings are cached for 24 hours so licence lookups don't hit the User Service every time.

📊 Analytics Service
Go · Port 8006
DB: analytics_db + Redis

Records every booking event to both Postgres (permanent audit trail) and Redis counters (real-time stats). An hourly rollup goroutine aggregates the previous hour's data into a summary table. It also serves as the system's health hub — its /api/analytics/health/services endpoint probes all six services in parallel and returns a combined status report. Postgres replica lag is exposed at /api/analytics/replica-lag.

Why split into six services?

Each service can be updated or restarted independently. A bug in Analytics doesn't affect booking. Conflict detection (which needs Go's goroutines for tight locking) can be scaled separately from User registration (which is write-infrequent). Teams can work in parallel without stepping on each other's code.

3Infrastructure

Behind the six services sits a supporting layer of databases, caches, a message broker, and load balancers. Here's what each piece does and why it was chosen.

PostgreSQL — one database per service

Each service has its own Postgres database running as a primary + replica pair. The primary handles all writes; the replica handles reads. This spreads load and means the replica can serve enforcement/analytics queries without slowing down booking writes.

The Conflict service uses SERIALIZABLE isolation for its check-and-reserve operation. All other services use the default READ COMMITTED — SERIALIZABLE everywhere would cause unnecessary lock contention for simple CRUD.

Redis + Sentinel — fast cache with automatic failover

Redis is used for three things: (1) the Enforcement service's vehicle lookup cache (sub-millisecond reads), (2) message deduplication keys in consumers (Redis SETNX), and (3) Notification history lists per user.

Three Sentinel instances watch the Redis primary. If the primary dies, they vote (quorum = 2-of-3) and promote the replica automatically, usually within ~15 seconds. Services auto-reconnect — there's no human intervention needed.

RabbitMQ — async event delivery

After a booking is confirmed, four things need to happen: the driver gets notified, the analytics counters update, the enforcement cache gets populated, and the conflict slot gets replicated. We don't want any of these to slow down the booking response. So they're done asynchronously via a RabbitMQ topic exchange called journey_events.

Each consumer has its own durable queue. If a consumer crashes, messages queue up and are delivered when it restarts — nothing is lost. Each queue also has a dead-letter exchange (DLX): if processing a message fails repeatedly, the message gets moved to a dead-letter queue where it sits for 24 hours for manual inspection.

HAProxy + nginx — the gateway

All browser traffic enters through HAProxy on port 8080, which load-balances across two nginx instances (in full-stack mode). nginx handles URL routing (which request goes to which service), rate limiting, and per-request DNS resolution.

Rate limits: authentication endpoints (5 req/s, burst 10), booking endpoints (10 req/s, burst 20), general (30 req/s). Per-request DNS is important — without it, nginx caches container IPs and breaks after a container restart.

4How a Booking Works — The Saga Pattern

When a driver books a journey, the Journey service orchestrates a sequence of steps called a saga. Each step either succeeds and moves forward, or fails and the whole thing is rejected. Here's exactly what happens:

  • 1
    Idempotency check

    The driver's request includes an idempotency_key. If this exact key has been seen before, the Journey service returns the cached result immediately — no processing. This means a driver can retry a request after a network failure without accidentally creating two bookings.

  • 2
    Insert journey with status PENDING

    A journey row is created in the database with status PENDING before anything else happens. This creates a record even if the service crashes mid-way through the saga.

  • 3
    Call the Conflict service to check and reserve the slot

    The Journey service calls POST /api/conflicts/check on the Conflict service. This call is wrapped in a circuit breaker — if the Conflict service has been failing, the circuit opens and the booking fails fast instead of hanging for 30 seconds. If the local Conflict service is down, it automatically tries peer nodes on other laptops.

    Inside the Conflict service, a single SERIALIZABLE transaction runs three checks: driver overlap, vehicle overlap, and road capacity. If anything conflicts, it returns REJECTED with a reason. If everything is clear, it reserves the slot.

  • 4
    Write journey + outbox event in a single database transaction

    If the conflict check passed, the Journey service updates the journey to CONFIRMED and writes an outbox event row — both in the same database transaction. Either both are committed or neither is. This is the key to the outbox pattern (explained in the next section).

  • 5
    Return CONFIRMED to the driver (~100–250ms)

    The synchronous part of the saga is done. The driver's browser shows "Booking confirmed." Everything after this happens in the background.

  • 6
    Background: outbox drainer publishes the event to RabbitMQ

    A background task polls the outbox_events table every 2 seconds for rows where published = false. It publishes each one to RabbitMQ and marks it published = true. If RabbitMQ is down, rows accumulate and drain when it comes back up.

  • 7
    Downstream services consume the event

    Notification service pushes a WebSocket message to the driver's browser. Analytics service increments its counters. Enforcement service adds the booking to its Redis cache. Each consumer deduplicates using Redis SETNX — if RabbitMQ redelivers the same message, it's silently ignored.

What about Two-Phase Commit (2PC)?

The system also supports a ?mode=2pc query parameter that switches to a stricter protocol. In 2PC, the PREPARE phase tentatively reserves the slot without committing. Only if the Journey service can successfully complete its own write does it send COMMIT. On any failure, it sends explicit ABORT to release the slot. This gives stronger atomicity but is slower — the slot is held between PREPARE and COMMIT. The saga mode is the default because it's faster and simpler, and 2PC is available to demonstrate the concept.

5Conflict Detection

The Conflict service's job is to answer one question: "can this booking proceed without conflicting with an existing one?" It must answer correctly even if two drivers are booking simultaneously.

The three checks

① Driver overlap

Does this driver already have a CONFIRMED journey that overlaps in time with the requested slot?

② Vehicle overlap

Is this vehicle registration already assigned to a CONFIRMED journey that overlaps in time?

③ Road capacity

Is the road segment at maximum capacity for this time slot? The road is divided into a grid of ~1km cells. The booking's route is traced through the grid, and each traversed cell's slot counter is checked and incremented. If any cell is full, the booking is rejected.

Why SERIALIZABLE + SELECT FOR UPDATE?

All three checks run inside one SERIALIZABLE database transaction with SELECT FOR UPDATE. This is intentional and important.

Consider the alternative: with the default READ COMMITTED isolation, two concurrent bookings for the same slot could both read "capacity = 1 remaining", both decide they're fine to proceed, and both commit — resulting in over-capacity. SERIALIZABLE prevents this: the second transaction either sees the first one's reservation as already committed (and rejects), or it detects a conflict and retries.

The cost of SERIALIZABLE

Higher lock contention than READ COMMITTED. However, we only use SERIALIZABLE in the Conflict service — all other services use the default. The lock scope is narrow (just the grid cells the route passes through), and conflict checks are fast (~5ms), so contention is minimal in practice.

The grid-cell road model

The road network is approximated as a grid with 0.01° resolution — roughly 1km per cell at Ireland's latitude. Each cell has a capacity limit per 30-minute time slot. When a booking is made, its route is traced through the grid and each traversed cell's counter is incremented atomically.

This design exploits spatial locality: two bookings on physically separate roads will traverse different cells and never contend, even if they're scheduled for the same time. False conflicts (two routes that are geographically close but physically separate) are avoided by using the actual road waypoints to trace cells, not just straight-line interpolation.

6The Transactional Outbox Pattern

This pattern solves a fundamental distributed systems problem: how do you update a database and send a message without losing one if the service crashes between the two?

The problem it solves

The naive approach: (1) write CONFIRMED to the database, (2) publish to RabbitMQ. But what if the process crashes after step 1 and before step 2? The booking is confirmed in the DB, but Notification never fires, Enforcement cache is never populated, Analytics never counts it. Silent inconsistency — no error, just missing data downstream.

The solution

Instead of publishing to RabbitMQ directly, we write an event row to an outbox_events table in the same database transaction as the journey row:

BEGIN TRANSACTION
  UPDATE journeys SET status = 'CONFIRMED' WHERE id = ?
  INSERT INTO outbox_events (routing_key, payload, published) VALUES ('journey.confirmed', {...}, false)
COMMIT

Either both rows are written or neither is. The outbox event is now durable — it survives any crash. A separate background task polls the outbox table every 2 seconds for rows where published = false, publishes them to RabbitMQ, and marks them published = true.

What happens during a RabbitMQ outage

Bookings continue to be confirmed normally. Outbox rows accumulate in the database with published = false. The driver gets their confirmation immediately. When RabbitMQ comes back up, the background drainer publishes everything that accumulated — Notification, Analytics, and Enforcement all catch up automatically. No manual intervention needed.

At-least-once delivery

If the drainer publishes an event but crashes before marking it published = true, the event will be published again on restart. This means messages can be delivered more than once. Every consumer handles this by doing a Redis SETNX check on the message ID — if the key already exists, the message was already processed and is silently acknowledged.

What could be improved

The 2-second polling interval adds latency between booking confirmation and downstream updates. A better approach would use PostgreSQL LISTEN/NOTIFY — the database would push a signal to the drainer the instant a new outbox row is inserted, reducing lag from ~2 seconds to milliseconds.

7Real-Time Notifications

After a booking event is published to RabbitMQ, the Notification service picks it up and pushes it to the driver's browser via WebSocket. The connection is established when the driver logs in and stays open as long as the tab is open.

The WebSocket registry is a simple in-memory map: user_id → [list of open connections], protected by a read-write mutex. If a driver has multiple browser tabs open, all of them receive the notification simultaneously.

Notification history

Notifications are also stored in Redis as a capped list per user: max 50 entries, 7-day TTL. Each write is a pipelined LPUSH (add to front) + LTRIM (trim to 50) + EXPIRE (reset TTL). Drivers can fetch their history via GET /api/notifications/ in case they missed live notifications.

Known limitation: in-memory connection registry

If the Notification service restarts, all WebSocket connections drop and clients must reconnect. There's no shared connection state. The Redis history ensures no notification content is lost, only the live push. To scale Notification horizontally, you'd need Redis Pub/Sub as a shared fan-out mechanism instead of an in-memory map.

8Enforcement Verification

Enforcement agents need to check a vehicle plate or driving licence at the roadside — quickly. The target is under 200ms, and ideally under 20ms. This is achieved with a two-layer lookup.

Layer 1: Redis cache (sub-20ms)

When a booking is confirmed, the Journey service (via the outbox + RabbitMQ) triggers the Enforcement service to store the booking in Redis under two keys: active_journey:vehicle:{plate} and active_journey:user:{user_id}. The TTL is set to the journey's estimated arrival time + 1 hour.

On an enforcement check, Redis is hit first. If the entry exists and the journey is currently within its time window (departure ≤ now + 30 min, arrival ≥ now), it's returned immediately.

Layer 2: Journey Service REST fallback

On a cache miss (cache cold, TTL expired, or Redis down), the Enforcement service calls GET /api/journeys/vehicle/{plate}/active on the Journey service. On a successful fallback hit, the cache is populated for future requests.

Licence lookups

When an agent queries by driving licence number, the service first resolves the licence to a user ID (via the User service), then looks up the user's active journey. The licence→user_id mapping is cached in Redis for 24 hours so the User service isn't hit on every check.

During a network partition

If the Journey service is unreachable, enforcement continues from the Redis cache. Responses include X-Cache-Stale: true and X-Partition-Status: journey-service: PARTITIONED headers to alert the agent that the data may be slightly outdated. This is a deliberate availability-over-consistency tradeoff for roadside enforcement — showing a stale "valid" result is safer than blocking the check entirely.

Known limitation: cold cache on startup

After a service restart, the Redis cache is empty. Every enforcement check pays the full REST fallback cost until the cache warms up via incoming booking events. There's a manual endpoint (POST /admin/recovery/rebuild-enforcement-cache) that rebuilds the cache from the DB, but it's not called automatically on boot.

9Circuit Breaker

When the Conflict service is down, we don't want booking requests to hang for 30 seconds waiting for a timeout. The circuit breaker fails fast — after a few failures it stops trying for a while, protecting the system from cascading slowdowns.

  • ●CLOSED — Normal operation. All calls go through. Failure counter increments on each error.
  • ●OPEN — After 3 consecutive failures, the circuit opens. All calls return an error immediately without making any network request. After 30 seconds, moves to HALF-OPEN.
  • ●HALF-OPEN — One probe call is allowed through. If it succeeds, circuit closes (recovered). If it fails, circuit re-opens for another 30 seconds. This prevents hammering a service that's still recovering.

Per-URL circuit breakers

We have one independent circuit breaker per endpoint URL, not a single global one. This is critical: if the local Conflict service crashes, its circuit opens — but the circuit breaker for the peer Conflict service on Laptop B is unaffected. Booking requests automatically route to the peer, which still has a fresh, closed circuit.

The failover chain in conflict_client.py:

1. Try local conflict-service  (circuit: CLOSED → try it)
2. Try peer conflict-service on Laptop B  (circuit: CLOSED → try it)
3. Try any other registered peers...
✗  All unreachable → booking REJECTED
What could be improved

When all Conflict service nodes are unreachable, the booking is simply rejected. A better design would queue the booking and retry automatically once a circuit closes — a "retry saga" pattern. Implementing this requires tracking in-progress bookings and compensating actions, which we documented as a known gap.

10Redis Sentinel — Cache High Availability

Redis is used for enforcement caching and deduplication — both of which are on the hot path of user-facing requests. If Redis just had a single instance, a crash would mean every enforcement check falls back to the REST API (slower) and message deduplication stops working (risk of double-processing).

Redis Sentinel addresses this with automatic failover:

  • ✓Three Sentinel processes run alongside Redis, continuously monitoring the primary
  • ✓If the primary stops responding, Sentinels vote — 2 out of 3 must agree before a failover starts (quorum)
  • ✓The winning Sentinel promotes the replica to primary, usually within ~15 seconds
  • ✓It updates the Redis replication topology and notifies connected clients
  • ✓Application services reconnect automatically using the Sentinel discovery protocol
Why quorum 2-of-3?

With only 1 Sentinel, a network partition where the Sentinel can't reach Redis (but Redis is actually fine) would trigger a spurious failover. With 3 Sentinels requiring 2 votes, a single sentinel being disconnected isn't enough to trigger failover — at least 2 must independently agree that the primary is unreachable.

11PostgreSQL Streaming Replication

Each of the four service databases (users, journeys, conflicts, analytics) runs a primary + replica pair using WAL (Write-Ahead Log) streaming replication.

WAL is Postgres's built-in mechanism for durability: before any change is applied, it's first written to the WAL. In streaming replication, the replica subscribes to the primary's WAL stream and applies changes in near real-time. Replica lag is typically under 100ms on the same machine.

How we use it

Services maintain two database session pools: one pointing at the primary (for writes), one pointing at the replica (for reads). The appropriate pool is selected via dependency injection in each route handler. This means read-heavy endpoints (listing journeys, checking analytics) don't compete with write operations for the same database connections.

The Analytics service exposes replica lag at /api/analytics/replica-lag by querying pg_stat_replication on each primary. This provides live proof that replication is active.

When the replica lags or crashes

If replica lag exceeds a threshold, reads fall back to the primary automatically (implemented at the application level). If the replica crashes entirely, the service continues using the primary for both reads and writes — there's no automated promotion of the replica to primary (unlike Redis Sentinel). A proper production system would use Patroni or similar for Postgres HA, but for this demo, manual recovery is acceptable.

12Multi-Node Design — Active-Active across laptops

In the multi-laptop setup, each laptop runs the complete stack. Both nodes accept bookings and serve all traffic. There's no designated "master" — this is active-active. Losing one node doesn't take the system down; the other node continues serving requests.

How nodes discover each other

Peer discovery uses two mechanisms:

  • ✓Static (at startup): Peer IPs are set in .env via PEER_CONFLICT_URLS and PEER_USER_URLS before starting Docker Compose. These are loaded when services start.
  • ✓Dynamic (at runtime): Call POST /admin/peers/register with the peer's IP. This propagates to the Journey service (for data replication), User service (for distributed locking), and Conflict service (for failover) — all without restarting anything.

The known gap: millisecond-window double booking

If two drivers submit bookings for the exact same slot at almost exactly the same time — one to each laptop — both bookings can be CONFIRMED. Laptop A's Conflict service hasn't yet received Laptop B's slot replication. Without a consensus protocol (Raft, Paxos), this window cannot be fully closed. At demo scale and realistic booking patterns, this rarely occurs. A production system would use a consensus protocol or route all writes for a given route to a single "home" node.

13Client-Side Failover

Every API call from the browser goes through a JavaScript function called resilientFetch. This function knows about all available peer nodes and retries automatically on failure:

  1. 1
    Try the current primary URL

    Usually the user's own laptop's HAProxy at http://<my-ip>:8080.

  2. 2
    On 5xx or network error: try the next peer

    4xx responses (like "booking rejected") pass through unchanged — a rejected booking isn't a server failure. Only server errors trigger failover.

  3. 3
    The peer list is fetched from /health/nodes at login

    It's stored in localStorage, so the browser can failover even before the next login — the peer list survives page refreshes.

Because all nodes share the same JWT signing secret, a token issued on Laptop A is valid on Laptop B. The user is never asked to log in again after a failover — the session migrates transparently.

The WebSocket connection to Notification service uses the same peer list, cycling to a peer after two consecutive disconnects.

Simulating a node failure

Calling POST /admin/simulate/fail causes the Journey service and User service to return 503 on all requests. The browser detects this on the next request and switches to the peer's URL, showing "Failover: <peer-ip>" in the navigation bar. Calling POST /admin/simulate/recover restores both services, and the next health check from the peer marks this node ALIVE again.

14Design Choices & Why

DecisionWhat we choseWhyWhat we gave up
Booking protocol Saga (default), 2PC optional Saga is faster and works even if the Conflict service restarts mid-flow. 2PC is stricter — slot held between PREPARE and COMMIT — but blocks until coordinator decides. Saga has a gap: if the Journey service crashes after conflict approval but before DB write, the slot may be held unnecessarily until it expires.
Conflict isolation SERIALIZABLE + SELECT FOR UPDATE, Conflict service only Only true way to prevent double-booking under concurrent load. READ COMMITTED allows two transactions to both see "1 slot remaining" and both commit. Higher lock contention in the Conflict service. Acceptable because the check is fast and lock scope is narrow (one grid cell).
Event delivery Transactional outbox + RabbitMQ Outbox ensures no event is ever lost — it's atomic with the DB write. RabbitMQ decouples producers from consumers; downstream services don't need to be up for a booking to succeed. 2-second polling delay. More operational complexity. Downstream services lag during RabbitMQ outages (though they catch up automatically).
Multi-node consistency Active-active eventually consistent Raft/Paxos adds significant complexity, requires a dedicated leader election round on every write, and introduces leader-election latency. For a 2-laptop demo with low booking rates, eventual consistency is sufficient. Millisecond window where two nodes can both confirm the same slot. Documented and accepted as a known trade-off.
Cache strategy TTL eviction + event-driven invalidation TTL handles the normal case (journey expires naturally). Event-driven invalidation (on cancel/complete) handles early removal without a full cache flush or expensive cache-aside logic. Cache is cold on startup. First enforcement check always misses. No automatic cache warming on boot.
Service languages Python (User, Journey, Enforcement) + Go (Conflict, Notification, Analytics) Python (FastAPI) is faster to write complex orchestration code. Go handles concurrent operations better — thousands of WebSocket connections in Notification, tight mutex locking in Conflict, concurrent counter updates in Analytics. Two languages means two codebases. Go services can't share the Python shared/ library — equivalent packages had to be written separately.
Sharding MD5 modulo, write-authority only All nodes hold all data (active-active). Sharding only governs which node is the "authoritative writer" for a given email or route ID. Simple to implement and reason about. Modulo hashing (not consistent hashing) — adding a node changes shard assignments for all existing keys. For 2-3 nodes this is acceptable; for dynamic scaling, a consistent hash ring would be needed.
Distributed lock Redlock-style 2-phase (SETNX on each peer) Prevents two nodes from simultaneously accepting the same email address. Simpler than a full Redlock implementation while providing equivalent guarantees at 2-3 node scale. If a peer is unreachable during registration, the lock request fails conservatively (rejects). Network partition during registration causes 409 even if the email is actually unique.

15Known Limitations

These are documented honestly — deliberate trade-offs accepted for the scope of this project, not hidden bugs.

No consensus protocol → possible double booking

Two simultaneous bookings on two different nodes within the replication window can both be confirmed for the same slot. A Raft or Paxos-based system would route all writes through a single elected leader, eliminating this. At demo scale and realistic booking rates, this window is rarely hit.

No saga compensation retry

If the Conflict service is unreachable, the booking fails immediately. There's no retry queue or compensation loop. A production saga would enqueue the retry and attempt again once the circuit closes.

JWT tokens not blacklisted on logout

A revoked token remains valid until its natural expiry. Logout is client-side only (token removed from browser storage). A production system would maintain a Redis deny-list of invalidated token IDs.

WebSocket registry is in-memory

If the Notification service restarts, all WebSocket connections drop. Clients must reconnect. Notification content is preserved in Redis history, but the live push is lost. Horizontal scaling would require Redis Pub/Sub fan-out.

Enforcement cache cold on startup

After a restart, the Redis cache is empty. A manual rebuild endpoint exists (POST /admin/recovery/rebuild-enforcement-cache) but it's not called automatically. The first enforcement check for every vehicle pays the full REST fallback cost.

Outbox polls every 2 seconds

Downstream services (Notification, Analytics, Enforcement) lag up to 2 seconds behind booking confirmation. PostgreSQL LISTEN/NOTIFY would reduce this to near-zero latency.

No distributed tracing UI

Correlation IDs (X-Request-ID) propagate across all services and appear in every log line. But there's no Jaeger or Zipkin integration to visualise traces. Cross-service latency debugging requires grepping logs manually.

RabbitMQ cluster unreliable on a single host

A 3-node RabbitMQ cluster is configured, but Erlang distribution is unreliable when all three nodes run on the same Docker host (networking and node naming conflicts). The cluster works correctly across physical machines. Single-machine demos effectively run a single broker.

What the system does well

No confirmed booking is ever lost (transactional outbox). Single node failure is transparent to users (shared JWT, resilientFetch). Conflict detection is atomic and correct under concurrent load (SERIALIZABLE). Enforcement checks are fast from cache. Circuit breakers prevent cascading failures. The entire system is observable — every cross-service call carries a correlation ID, partition status is exposed on every response, and replica lag is monitored.
